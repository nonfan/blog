最直观的区别便是网址前缀 `http://` 和 `https://` 的区别，但其中蕴含的问题就没那么简单了，`https` 更像是安全体系中的护城河，用来抵御传输过程中的攻击者。

## HTTP 是什么？它为什么不安全？

HTTP 的本质：明文传输协议

HTTP（HyperText Transfer Protocol）本质上只是一个应用层协议，它做的事情非常简单：

> 浏览器把请求发给服务器
> 服务器把内容原样返回

整个过程不加密、不校验、不验证身份。

也就是说，你在 HTTP 网站上发送的数据，看起来大概是这样：

```txt
GET /login
username=alice
password=123456
```

**明文传输意味着什么？**

意味着在“你和服务器之间”的任何一环如：

- 公共 WiFi
- 路由器
- 运营商
- 被劫持的网关
- 抓包工具

都可以看到你发了什么、修改你收到的内容、或者替换页面或脚本。

HTTP 就像一张明信片：
内容对所有“路过的人”都是可见的。

## HTTPS 是什么？它解决了哪些问题？

HTTPS 并不是一种全新的协议，而是：

HTTP 运行在 TLS（以前叫 SSL）加密通道之上

TLS 给 HTTP 增加了三项关键能力：

- 加密（Encryption）
- 身份认证（Authentication）
- 完整性校验（Integrity）

## HTTPS 安全的三大核心原因

### 加密
> 别人看不懂你在传什么

在 HTTPS 中：

> 浏览器和服务器会先协商一个对称密钥
> 后续所有数据都用这个密钥加密

### 身份认证
> 这一步解决的是“你连的是谁”的问题

**HTTPS 中，服务器必须出示一个数字证书：**

- 由受信任的 CA（证书颁发机构）签发
- 绑定域名和公钥

**浏览器会自动验证：**

- 证书是否可信
- 域名是否匹配
- 是否过期或被吊销

**如果验证失败：**

- 浏览器直接报警
- 用户无法“无感继续”

### 完整性校验

> 内容不能被偷偷篡改

HTTPS 使用 MAC / AEAD 等机制：

- 每一段数据都有校验信息
- 任何中途修改都会导致校验失败

结果是：

- 页面被改 → 连接断开
- JS 被注入 → 校验失败

## HTTP 和 HTTPS 的核心区别对比

| 对比项        | HTTP | HTTPS |
| ---------- | ---- | ----- |
| 是否加密       | ❌ 明文 | ✅ 加密  |
| 是否验证服务器身份  | ❌    | ✅     |
| 是否防篡改      | ❌    | ✅     |
| 是否防监听      | ❌    | ✅     |
| 浏览器信任度     | 低    | 高     |
| 是否被现代浏览器限制 | 是    | 否     |
